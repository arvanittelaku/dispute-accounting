# How the sitemap and robots.txt are created

**Site:** https://www.disputeaccounting.com  
This document describes how `public/sitemap.xml` and `public/robots.txt` are produced, where the URL list comes from, and how the project keeps them in sync with the app.

## Overview

Both files are **generated by a Node script**, not maintained by hand. That keeps every public URL aligned with the same inventory the verification tooling uses.

| Output | Path | Generator |
|--------|------|-----------|
| XML sitemap | `public/sitemap.xml` | `scripts/generate-seo.ts` |
| robots.txt | `public/robots.txt` | `scripts/generate-seo.ts` |

## URL inventory (`buildPublicUrlInventory`)

The script imports `buildPublicUrlInventory` from `src/lib/seo/publicUrlInventory.ts`. That function assembles all paths that should appear in search, then maps them to absolute URLs with the canonical host `https://www.disputeaccounting.com` (or `NEXT_PUBLIC_SITE_URL` when set).

The inventory merges:

1. **Static app routes** ‚Äî Curated list `APP_STATIC_PATHS` (home, services, dispute-types, who-we-help, case-types hub, sectors, guides, FAQ, glossary, etc.).
2. **Case type pages** ‚Äî From `src/data/case-types.ts` as `/case-types/:slug` (10 pages).
3. **Sector pages** ‚Äî From `src/data/sectors.ts` as `/sectors/:slug` (6 pages).
4. **Guide pages** ‚Äî From `src/data/guides.ts` as `/guides/:slug` (6 pages).

The combined list is **deduplicated**, **sorted**, and exposed as `allUrls` (absolute) and `allPaths` (pathname only).

**Excluded from sitemap** (`SITEMAP_EXCLUDED_PATHS`):

- `/contact` ‚Äî indexable but omitted from sitemap (conversion page)
- `/thank-you` ‚Äî `noindex, nofollow`
- `/privacy` ‚Äî `noindex, follow`
- `/terms` ‚Äî `noindex, follow`

## Sitemap generation (`scripts/generate-seo.ts`)

For each URL in `inventory.allUrls`, the script emits a standard sitemaps.org `<url>` entry with:

- `<loc>` ‚Äî The absolute URL
- `<lastmod>` ‚Äî Calendar date when generation runs (`YYYY-MM-DD` in UTC)
- `<changefreq>` ‚Äî Heuristic by path (home `weekly`, others `monthly`)
- `<priority>` ‚Äî Heuristic by path (see `docs/SEO-ARCHITECTURE.md` Appendix B)

The full document is written to `public/sitemap.xml`. The `public/` directory is created if needed.

## robots.txt generation (`scripts/generate-seo.ts`)

`robots.txt` is a **fixed template** in the same script (`renderRobots()`), not derived from the URL inventory. It:

- Allows all crawlers on `/` by default
- Disallows `/thank-you`, `/api/`, and `/_next/`
- Declares the sitemap: `Sitemap: https://www.disputeaccounting.com/sitemap.xml`

The file is written to `public/robots.txt`.

## Commands

From the repo root:

```bash
npm run seo:generate   # Regenerate sitemap.xml + robots.txt
npm run seo:verify     # Fail if sitemap <loc> entries drift from inventory
```

`npm run build` runs `seo:generate` before `next build`.

Run `seo:generate` after:

- Adding a path to `APP_STATIC_PATHS`
- Adding slugs to case-types, sectors, or guides data
- Changing robots rules in `renderRobots()`
- Before a release when you want fresh `lastmod` dates

## Continuous integration

`.github/workflows/seo-checks.yml` runs on pull requests:

1. `npm ci`
2. `npm run seo:generate`
3. `npm run seo:verify`

## Deployment (Netlify)

`netlify.toml` sets explicit content types:

- `/sitemap.xml` ‚Üí `Content-Type: application/xml`
- `/robots.txt` ‚Üí `Content-Type: text/plain`

Files in `public/` are served from the site root at:

- https://www.disputeaccounting.com/sitemap.xml
- https://www.disputeaccounting.com/robots.txt

## Operational notes

- **Canonical host** is defined in `publicUrlInventory.ts` (`CANONICAL_HOST`) and in the robots `Sitemap:` line. If the primary domain changes, update both and regenerate.
- **New static pages** must be added to `APP_STATIC_PATHS` or `seo:verify` will fail after the next generate/commit drift check.
- This site has **no dynamic blog/API URL sources** (unlike multi-source inventories on larger properties). All URLs come from static paths + three data files.
